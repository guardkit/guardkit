# Review Report: TASK-REV-5B3A

## Executive Summary

Running `guardkit graphiti seed --force` from the `agentic-dataset-factory` directory seeded **GuardKit's own system context** (product_knowledge, command_workflows, quality_gate_phases, etc.) instead of the project-specific architecture artefacts. This is **expected behaviour by design** — the `seed` command is explicitly for seeding GuardKit system context. The project-specific architecture artefacts were never intended to be seeded by this command and require `guardkit graphiti add-context` instead.

## Finding 1: `guardkit graphiti seed` Seeds GuardKit System Context By Design

**Severity**: Informational (not a bug)

**Evidence**: The `_cmd_seed()` function in [graphiti.py:100-262](guardkit/cli/graphiti.py#L100-L262) calls `seed_all_system_context()` from [seeding.py:107-221](guardkit/knowledge/seeding.py#L107-L221), which orchestrates 17 hardcoded categories of **GuardKit knowledge**:

| Category | Content |
|----------|---------|
| product_knowledge | GuardKit mission, features |
| command_workflows | /task-work, /task-review etc. |
| quality_gate_phases | Phase 2.5, 4.5 etc. |
| technology_stack | Python, CLI architecture |
| feature_build_architecture | AutoBuild Player-Coach |
| architecture_decisions | Feature-build ADRs (ADR-FB-001..003) |
| failure_patterns | Known failure modes |
| component_status | Component readiness |
| integration_points | MCP, Conductor etc. |
| templates | Stack templates (filtered by template) |
| agents | Agent definitions |
| patterns | Code patterns |
| rules | Rule files |
| project_overview | **GuardKit** overview (guardkit_mission, guardkit_commands_overview, etc.) |
| project_architecture | **GuardKit** directory layout, Conductor integration |
| failed_approaches | Known anti-patterns |
| pattern_examples | Code examples |

The `project_overview` and `project_architecture` categories (added by TASK-CR-005) seed **GuardKit's own** project info — hardcoded in [seed_project_overview.py](guardkit/knowledge/seed_project_overview.py) and [seed_project_architecture.py](guardkit/knowledge/seed_project_architecture.py). They do NOT read from the current project's files.

**Conclusion**: The command worked exactly as designed. The confusing naming (`project_overview`/`project_architecture`) makes it look like it should seed the current project, but it seeds GuardKit's own context.

## Finding 2: Duplicate Knowledge Impact (Two Seed Runs)

**Severity**: Low

**Evidence**: Two seed markers exist:
- `.guardkit/seeding/.system_seeded.json` (2026-03-16, template: langchain-deepagents, 6 episodes) — from `seed-system` command
- `.guardkit/seeding/.graphiti_seeded.json` (2026-03-17, v1.2.0) — from `seed` command with `--force`

These are **different commands** with **different markers**:
- `seed-system` → `.system_seeded.json` (Phase 2 seeding: templates, roles, modes)
- `seed` → `.graphiti_seeded.json` (Legacy seeding: all 17 categories)

**Deduplication mechanism**: The `upsert_episode()` method in [graphiti_client.py:1280-1425](guardkit/knowledge/graphiti_client.py#L1280-L1425) uses `entity_id` + `source_hash` for idempotency:
1. Checks if episode with matching `entity_id` exists
2. If exists with same `source_hash`: **skips** (no duplicate)
3. If exists with different `source_hash`: **updates** in place

**Impact assessment**:
- Categories seeded by both commands (templates, agents, rules) would have been **deduplicated** by `upsert_episode()` because they share the same `entity_id` values
- The `seed` command's additional categories (product_knowledge, command_workflows, etc.) are unique to that command, so no duplicates there
- The Graphiti LLM warnings (`LLM returned invalid duplicate_facts idx values`) visible in the seed output indicate Graphiti's own deduplication layer was also active and handling overlaps

**Conclusion**: **Minimal duplicate impact**. Graphiti's upsert mechanism prevents true duplicates. At worst, some nodes/edges may have been re-processed (updated with identical content), consuming LLM tokens without creating redundant data.

## Finding 4: The Correct Command Already Exists

**Severity**: Informational

The `guardkit graphiti add-context` command ([graphiti.py:821-891](guardkit/cli/graphiti.py#L821-L891)) is the correct mechanism for seeding project-specific artefacts:

```bash
# Seed architecture docs
guardkit graphiti add-context docs/architecture/ --type architecture

# Seed ADRs specifically
guardkit graphiti add-context docs/architecture/decisions/ --type adr

# Seed all docs
guardkit graphiti add-context docs/architecture/ --pattern "**/*.md"
```

The `seed` command itself even prints this hint on completion (line 256-257):
```
To seed project ADRs:
  guardkit graphiti add-context docs/adr/ --type adr
```

**Gap**: The hint assumes ADRs are at `docs/adr/`, but `/system-arch` generates them at `docs/architecture/decisions/`. This path mismatch may have contributed to the user not running the correct command.

## Finding 5: Whether GuardKit System Context Is Useful in agentic-dataset-factory

**Severity**: Medium (decision required)

The GuardKit system context seeded under `project_id: agentic-dataset-factory` is **namespaced** (all group_ids prefixed with `agentic-dataset-factory__`). This means:

- It does NOT pollute a global namespace
- It IS queryable when working in the agentic-dataset-factory project
- It provides GuardKit command/workflow knowledge when using GuardKit features within the project

**Trade-off**:
- **Keep it**: Provides GuardKit operational knowledge (useful if using /task-work, /feature-build etc. in the project). Cost: ~80 episodes of storage in FalkorDB.
- **Remove it**: Cleaner namespace, only project-specific knowledge. But you'd lose GuardKit command reference within the project context.

## Recommendations

### Recommendation 1: Seed the Missing Architecture Artefacts (Priority: High)

Run from the `agentic-dataset-factory` directory:

```bash
# Seed all architecture documentation
guardkit graphiti add-context docs/architecture/ --pattern "**/*.md"

# Or separately for more control:
guardkit graphiti add-context docs/architecture/ARCHITECTURE.md
guardkit graphiti add-context docs/architecture/decisions/ --type adr
guardkit graphiti add-context docs/architecture/domain-model.md
guardkit graphiti add-context docs/architecture/container.md
guardkit graphiti add-context docs/architecture/system-context.md

# Also seed assumptions (YAML)
guardkit graphiti add-context docs/architecture/assumptions.yaml
```

### Recommendation 2: Leave GuardKit System Context in Place (Priority: Low)

The GuardKit system context is namespaced and provides useful operational knowledge. **No cleanup needed.** If storage is a concern, it can be cleared later with:

```bash
guardkit graphiti clear --group system  # If such command exists
```

### Recommendation 3: Fix Post-Seed Hint Path (Priority: Low)

The hint at [graphiti.py:256-257](guardkit/cli/graphiti.py#L256-L257) says `docs/adr/` but `/system-arch` generates ADRs at `docs/architecture/decisions/`. Consider updating to be more generic or detecting the actual ADR path.

### Recommendation 4: Document the Two-Command Workflow (Priority: Medium)

The confusion arose because the user expected `guardkit graphiti seed` to seed project-specific content. The graphiti-knowledge.md rule file does document the two-phase architecture, but the command naming is ambiguous. Consider:

- Adding a `--project` flag to `seed` that also runs `add-context` on `docs/architecture/`
- Or documenting more prominently that `/system-arch` output requires a manual `add-context` step
- Or having `/system-arch` automatically offer to seed its output to Graphiti

### Recommendation 5: No New Command Needed (Priority: Informational)

`guardkit graphiti add-context` already covers the project-specific seeding use case. The gap is **workflow documentation**, not missing functionality.
