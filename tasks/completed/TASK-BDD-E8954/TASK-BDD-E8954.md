# Task: BDD oracle wiring — task-work reads task-level features/*.feature and writes bdd_results

## Context

Follow-on from **TASK-REV-4D012** (AutoBuild Coach integration review). Report: `docs/reviews/TASK-REV-4D012-autobuild-coach-integration-review.md`, §6 R2.

The review's F2 finding: `.feature` files produced by `/feature-spec` reach `/feature-plan` as `--context` and are then **discarded**. `.guardkit/features/FEAT-*.yaml` has zero `.feature` references; no repo has `pytest_bdd` imports. The Gherkin is the natural oracle for feature-level behaviour, and it is currently thrown away after the planner.

This task closes the loop at the **task-level** only.

## Problem Statement

Gherkin scenarios generated by `/feature-spec` are not executed as tests. Coach cannot use them as a verification oracle. The artefact exists; the pipe is missing.

## Scope

### In-Scope (task-level BDD only)

1. New module `guardkit/orchestrator/quality_gates/bdd_runner.py`:
   - Takes a task's scope identifier, locates any `features/*.feature` files whose scenarios carry a matching tag (e.g. `@task:AUTH-001`), and runs `pytest --gherkin-terminal-reporter features/X.feature -m <tag>`.
   - Returns a structured `BDDResult` with three outcome states (**not two**): `scenarios_passed`, `scenarios_failed` (assertion failure — real Coach-blocking bug), and `scenarios_pending` (scenario exists in `.feature` but has no implemented step definition — scaffolding state, not a regression).

### Three-state outcome model — *do not collapse to pass/fail*

A `.feature` file can exist with no step definitions yet (common output of early `/feature-spec` work). pytest-bdd reports these scenarios as `pending`. Treating them as failures would mean the first jarvis run with unimplemented step scaffolding looks like "BDD broke the build" when nothing of the sort happened.

`BDDResult` MUST distinguish:

| State | Meaning | Coach treatment |
|---|---|---|
| `passed` | Step ran, assertion succeeded | Counts toward approval |
| `failed` | Step ran, assertion **failed** | **Blocks approval** — real Coach-rejectable signal |
| `pending` | Step definition **not yet implemented** | Surfaces in Coach feedback as "implement step X in tests/steps/…"; **does not block** approval on its own |

Coach approval rule becomes: `bdd_results.scenarios_failed == 0` (pending state is tolerated). Pending scenarios still appear in the Coach's feedback as actionable work, just without the red-X treatment.
2. `task-work` invocation: after Phase 4 tests, if the bdd_runner found matching scenarios, execute them and write results into `task_work_results.json` under a new `bdd_results` key.
3. `autobuild-coach.md` (and `CoachValidator`) updated to read `bdd_results.scenarios_failed == 0` as one more approval gate **when present**.
4. Stack support at launch: **Python via pytest-bdd only**. Verified: jarvis, forge, study-tutor are all Python.

### Scoping boundaries — critical (do NOT cross)

| Level | Owner | Scope |
|---|---|---|
| **Task-level** (THIS TASK) | **TASK-BDD-E8954** | Scenarios tagged to a single `TASK-XXX`, run inside that task's Coach validation. |
| Feature-level composition | **TASK-SMK-F703A** | Whole-feature `.feature` run between autobuild waves. |
| Assumption-gated | Future (not here) | `_assumptions.yaml` low-confidence scenarios that pause on first touch. |

If `task-work` ends up running an untagged whole-feature `.feature`, the scope has slipped — that is SMK's territory, not BDD's.

### Activation trigger — critical

**Activation is by artefact presence, NOT by frontmatter flag.**

- If `features/*.feature` exists AND contains scenarios tagged for this task's scope AND `pytest-bdd` is installable → run.
- If any condition is absent → silently skip; behaviour identical to today.

A frontmatter flag could be forgotten by the Player; operators would think BDD ran when it didn't. Presence-of-artefact is determined by what `/feature-spec` produced, so the behaviour is self-consistent.

### Out-of-Scope

- TypeScript (`cucumber-js`) or .NET (`SpecFlow`) support — separate follow-on if ever needed.
- Feature-level composition BDD (SMK-F703A territory).
- Teaching `/feature-spec` to emit task-scope tags — that's a natural follow-on if Gherkin adoption grows.

## Implementation Notes

- `pytest-bdd` integration: use `--gherkin-terminal-reporter` for parse-friendly output.
- Task-scope tag convention: `@task:<TASK-ID>` on the scenario. Document the convention in `feature-spec.md`.
- `BDDResult.failures` should include scenario name, failing step, and reason — enough for the Coach to surface in feedback.
- Depends on `TASK-AC-53445` only for sequencing (ship order: AC → BDD → SMK); no code dependency.

## Non-Goals / Guardrails

- **Do not add `autobuild.bdd_oracle: true` or any other activation flag.** Presence of the artefact is the trigger.
- **Do not run the whole-feature `.feature` here.** That's SMK-F703A.
- **Do not change Coach prompt structure beyond adding the new gate.** No role-boundary blur.
- **Do not collapse pending into failed.** Pending-step scenarios are expected during early feature development and must not be reported as Coach-blocking failures. The first jarvis run that ships with `/feature-spec` scaffolding but no implemented steps should report pending scenarios as informational, not as build breakage.
