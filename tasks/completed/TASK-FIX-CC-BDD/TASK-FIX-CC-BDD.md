# Task: Coach independent_tests must scope BDD step-defs runs by `@task:` tag

## Context

**From review:** [TASK-REV-CC40 (study-tutor)](/Users/richardwoollcott/Projects/appmilla_github/study-tutor/.claude/reviews/TASK-REV-CC40-review-report.md) — finding F-1.

The Coach validator's `independent_tests` path runs `pytest <every test file the player declared>` to verify task-specific tests pass. When one of those test files is the autogenerated **pytest-bdd glue** (e.g. `features/<feature>/test_<feature>.py`), pytest collects **every scenario in the matching `.feature` file**, not just the scenarios the current task owns.

In FEAT-39E1 the master feature file `nats-fleet-integration.feature` carried `@task:` tags for **9 different tasks** (PH1-002, -004, -005, -006, -008, -009, PH2-001, PH2-003, PH3-002, PH3-004, PH3-005). TASK-NATS-PH1-006 owned exactly one scenario. The Coach test command (`coach_turn_2.json`) was:

```
pytest features/nats-fleet-integration/test_nats_fleet_integration.py \
       tests/unit/cli/test_serve_nats.py \
       tests/unit/roles/test_registry.py \
       tests/unit/test_env_example_nats.py -v --tb=short
```

— no tag filter. pytest-bdd v8 emitted unbound-step scenarios as **FAILED** test functions (not pending), so 30+ scenarios meant for downstream tasks always failed. Coach's gate `bdd_results.scenarios_failed > 0` (and the independent_tests assertion `tests_passed`) rejected on a deterministic, retry-immune signal.

The autogenerated step-defs file documents the intended contract:

> "scenarios … tagged for a downstream task … will land with that task; their steps remain intentionally unbound here. They surface as `scenarios_pending` and are tolerated by the Coach gate (`scenarios_failed == 0`)."

The dedicated `bdd_runner.py` already enforces this contract — its module docstring (lines 19–25) explicitly states it runs only scenarios tagged `@task:<TASK-ID>`. But `coach_validator.independent_tests` bypasses `bdd_runner` and shells `pytest` directly on whatever .py files appear in `task_work_results.tests_written`, which destroys the task-scope guarantee.

## Problem Surface

[guardkit/orchestrator/quality_gates/coach_validator.py:794](../../guardkit/orchestrator/quality_gates/coach_validator.py) (and the surrounding `_run_independent_tests` / "Running independent tests via SDK / subprocess" path).

The current command is built from the literal contents of `task_work_results.tests_written`. There is no detection that some entries are pytest-bdd glue files, and therefore no tag-scoping is applied.

## Acceptance Criteria

- [ ] When `task_work_results.tests_written` contains a file under `features/**/test_*.py` (or a file that imports `pytest_bdd.scenarios`), `coach_validator` either:
  - (a) **Preferred:** delegates that file's execution to `bdd_runner.run_bdd_for_task()` (which is already task-tag-scoped), and runs the remaining non-BDD test files via the existing pytest path; OR
  - (b) Adds a tag-scope filter to the pytest invocation (`-k` matching against `task_id` snake-case, or pytest-bdd's tag selector if exposed) so only `@task:<TASK-ID>` scenarios are collected.
- [ ] Unbound-step scenarios from peer tasks are counted as `scenarios_pending` (or simply not collected), not `scenarios_failed` — matching the documented contract in `bdd_runner.py:19` and the autogenerated step-defs docstring.
- [ ] Regression test: a fixture feature file with N scenarios across M tasks (e.g. 5 scenarios, 3 tasks) runs through `coach_validator.independent_tests` for task X and reports only X's scenarios in the test result; peer-task scenarios do not surface as failures.
- [ ] Replay test against the FEAT-39E1 fixture (turn 2 inputs preserved at `.guardkit/worktrees/FEAT-39E1/.guardkit/autobuild/TASK-NATS-PH1-006/`): coach decision flips from `feedback (parallel_contention)` to `approved`.
- [ ] Existing tasks with no BDD test files are unaffected (no behaviour change).
- [ ] No new dependency added (use the existing pytest / pytest-bdd surface).

## Implementation Notes

**File to modify:** `guardkit/orchestrator/quality_gates/coach_validator.py` — the `_run_independent_tests` (or equivalent) builder that constructs the pytest argv around line 660–795.

**Detection heuristic** (option a, delegate to bdd_runner):
```python
from .bdd_runner import is_bdd_glue_file, run_bdd_for_task

bdd_files, plain_files = [], []
for f in tests_written:
    (bdd_files if is_bdd_glue_file(f) else plain_files).append(f)

if bdd_files:
    bdd_result = run_bdd_for_task(task_id=task_id, ...)  # already tag-scoped
if plain_files:
    plain_result = self._run_pytest_subprocess(plain_files, ...)
# combine results
```

`is_bdd_glue_file` would either grep for `pytest_bdd.scenarios(` / `from pytest_bdd import` (cheap text scan) or check whether the file's parent directory contains a `.feature` file. Lightweight, no AST parsing.

**Alternative (option b, tag filter):**
```python
task_id_safe = task_id.replace("-", "_")  # pytest-bdd encodes test names
cmd += ["-k", task_id_safe]
```
This relies on pytest-bdd's test-name encoding, which is brittle across versions. Option (a) is preferred.

**Counting fix:** `_parse_bdd_junit` (in `bdd_runner.py:239`) already distinguishes pending from failed via JUnit XML. The independent_tests path needs to call into the same parser when running BDD files; do not re-implement.
