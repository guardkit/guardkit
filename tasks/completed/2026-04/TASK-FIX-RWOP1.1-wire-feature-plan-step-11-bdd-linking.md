# Task: Wire /feature-plan Step 11 (BDD scenario linking) — remove runner-without-producer orphan

## Problem Statement

[TASK-REV-RWOP1](../../../docs/reviews/TASK-REV-RWOP1-runner-without-producer-orphan-sweep.md)
Finding #1 identified that **Step 11 of
[installer/core/commands/feature-plan.md](../../../installer/core/commands/feature-plan.md)**
— the step that invokes the `bdd-linker` subagent to auto-`@task:`-tag
BDD scenarios generated by `/feature-spec` — is a compound
runner-without-producer orphan:

- `installer/core/commands/lib/bdd_linking_phase.py:run_linking_phase`
  has **zero** non-test callers (only 14+ references in
  `tests/integration/feature_plan/test_bdd_linking.py`).
- The `bdd-linker` subagent file
  (`installer/core/agents/bdd-linker.md`) exists and works, but is
  unreachable because the only path to invoke it runs through a
  Python matcher-callback that no production code sets up.
- The spec prose instructs Claude to `from ... import run_linking_phase`
  and compose a matcher callback inline — but Claude is a
  slash-command interpreter, not a Python runtime.

This is load-bearing for R2 cohort evidence. If the step silently
does not run during [TASK-COH-RUN1](TASK-COH-RUN1-forge-and-study-tutor-cohort-run.md),
cohort `.feature` files ship un-`@task:`-tagged, the R2 BDD oracle
(`bdd_runner.run_bdd_for_task`) collects `_PYTEST_EXIT_NO_TESTS` on
every task, Coach approves silently, and the cohort report reads
"0 scenarios failed across N tasks" — indistinguishable from a real
pass. This is exactly the R1 contamination pattern TASK-FIX-3C9D
remediated, reproduced in R2 shape.

This task blocks [TASK-COH-RUN1](TASK-COH-RUN1-forge-and-study-tutor-cohort-run.md).

## Scope

### In-Scope

Pick ONE of the two remediation paths documented in TASK-REV-RWOP1
Finding #1 (they are functionally equivalent; the choice is a
maintenance-cost tradeoff). The design decision should be captured
in the `implementation_plan.md` under Phase 2:

**Path A — Claude-runtime invocation (preferred, lowest cognitive
load):**

1. Rewrite Step 11 prose in
   [feature-plan.md](../../../installer/core/commands/feature-plan.md)
   to use the same imperative `INVOKE Task(bdd-linker, ...)` pattern
   that Step 2 already uses for `clarification-questioner` — i.e.
   drop the Python `from ... import run_linking_phase` preamble and
   the matcher-callback block entirely. Have the `bdd-linker` agent
   itself (a) read the feature YAML + `.feature` files from the
   workspace, (b) produce the `MatchingRequest`/`TaskMatch[]`
   mapping, and (c) apply the mapping by editing the `.feature`
   files directly.
2. Update
   [installer/core/agents/bdd-linker.md](../../../installer/core/agents/bdd-linker.md)
   so the agent does the end-to-end work, not just the matching
   step. Previously the agent was one cog in a multi-cog pipeline;
   it now owns the whole Step 11.
3. Keep `bdd_linking_phase.run_linking_phase` as a reference
   implementation (for tests and for future deterministic wiring),
   but mark it as "not currently invoked from production" in the
   module docstring so nobody is misled again.

**Path B — Producer-script path (heavier but preserves
`run_linking_phase` as the deterministic source-of-truth):**

1. Create
   `installer/core/commands/lib/feature_plan_bdd_link.py` with two
   modes: `--prepare` (reads feature YAML + `.feature` files, emits a
   `MatchingRequest` JSON to stdout) and `--apply` (reads a
   `TaskMatch[]` JSON from stdin + `.feature` files, writes updated
   `.feature` files via `bdd_linker.apply_mapping`). Expose both as
   CLI subcommands of the same script.
2. Add the script to
   [installer/core/commands/bin-entries.txt](../../../installer/core/commands/bin-entries.txt)
   so it installs as `~/.agentecflow/bin/feature-plan-bdd-link`.
3. Rewrite Step 11 prose in `feature-plan.md` to:
   `Execute: python3 ~/.agentecflow/bin/feature-plan-bdd-link --prepare --feature-slug $SLUG`
   then `INVOKE Task(bdd-linker, ...)` with the JSON piped via
   `--matching-request-file`
   then `Execute: python3 ~/.agentecflow/bin/feature-plan-bdd-link --apply --feature-slug $SLUG --task-matches $AGENT_OUTPUT`.
4. This matches the exact shape TASK-FIX-3C9D used for R1 (imperative
   `Execute:` via `generate-feature-yaml`).

Either way, produce:

- Updated `feature-plan.md` Step 11 prose.
- (Path A only) Updated `bdd-linker.md` agent spec.
- (Path B only) New `feature_plan_bdd_link.py` CLI script + manifest
  entry + install.sh re-run verification.
- End-to-end test that drives Step 11 via the chosen path against a
  fixture `.feature` file + feature YAML, and asserts the `.feature`
  file ends up with `@task:` tags on each scenario. Place at
  `tests/integration/feature_plan/test_bdd_linking_end_to_end.py`.
  (Existing `test_bdd_linking.py` keeps its isolated-unit scope.)

### Out-of-Scope

- Refactoring the `bdd-linker` agent's matching heuristics (confidence
  threshold, ambiguity rules, etc.) — those are tested and stable.
- Changes to `bdd_linker.apply_mapping` internals.
- Changing the `.feature`-file output format or `@task:` tag
  convention.
- Retroactively tagging features in the cohort repos (forge,
  study-tutor). That's TASK-COH-RUN1's pre-flight.
- Wiring Step 11 into the `--quiet` / `--no-interactive` flag
  semantics beyond what the existing prose already specifies.

## Implementation Notes

- The two paths differ mainly in where the "matcher" runs: Path A
  hosts it inside the `bdd-linker` agent's own reasoning loop; Path B
  hosts it inside a Python CLI shim that the agent produces input
  for. For a project that uses Claude-as-runtime heavily, Path A is
  the cleaner match; for a project that wants deterministic
  command-line reproducibility, Path B is the cleaner match. TASK-
  FIX-3C9D chose Path-B-shape for R1 (AC linter) — worth considering
  the same choice here for consistency, but it isn't obligatory.
- The existing `bdd_linking_phase.run_linking_phase` contract is well
  tested (interactive / non-interactive / threshold / all-tagged
  idempotency / matcher-errors / low-confidence-skip). Don't rewrite
  the internals; only change who calls them.
- Per TASK-FIX-7B2E §"Retro grep of FEAT-JARVIS-001": dynamic
  verification in a single Claude session is NOT sufficient evidence
  that Claude-as-runtime will activate Step 11 reliably. If Path A is
  chosen, the same retro-grep caveat applies. The structural fix
  (Path B) is robust against Claude-session variance — worth factoring
  into the path decision.
- **TASK-COH-RUN1 dependency update**: after this task lands, add
  `TASK-FIX-RWOP1.1` to TASK-COH-RUN1's `depends_on` (RWOP1 review
  already notes this but the file edit is part of this task's
  closeout).
