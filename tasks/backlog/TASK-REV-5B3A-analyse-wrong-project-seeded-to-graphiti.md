# Task: Analyse wrong project seeded to Graphiti after install.sh fix

## Description

After running `install.sh` (to fix the graphiti-check environment issue from TASK-REV-4219), the user ran `guardkit graphiti seed --force` from the `agentic-dataset-factory` directory. The output (documented in `docs/reviews/additonal-templates/seed-graphiti.md`) shows that **GuardKit system context was seeded** rather than the agentic-dataset-factory project-specific architecture artefacts.

### Key Observations from Seed Output

1. **What WAS seeded**: GuardKit's own system context — product_knowledge, command_workflows, quality_gate_phases, technology_stack, feature_build_architecture, failure_patterns, component_status, integration_points, templates, patterns, rules, project_overview, project_architecture, failed_approaches, pattern_examples (~80+ episodes of GuardKit knowledge)

2. **What was NOT seeded**: The agentic-dataset-factory architecture artefacts that were generated by `/system-arch` but never persisted to Graphiti:
   - `docs/architecture/ARCHITECTURE.md` (system context, 6 modules/components)
   - `docs/architecture/decisions/ADR-ARCH-001` through `ADR-ARCH-009` (9 ADRs)
   - `docs/architecture/assumptions.yaml` (10 assumptions)
   - `docs/architecture/domain-model.md`
   - `docs/architecture/container.md`
   - `docs/architecture/system-context.md`

3. **Duplicate concern**: The `--force` flag was used, and a prior `.system_seeded.json` marker existed from an earlier seed (2026-03-16, template: langchain-deepagents). This means GuardKit system context may have been seeded **twice** into the agentic-dataset-factory namespace (project_id: `agentic-dataset-factory`), potentially creating duplicate nodes/edges.

4. **Project ID prefixing**: The graphiti config has `project_id: agentic-dataset-factory`, so all seeded episodes should have been prefixed with this project ID in their group_ids. This means the GuardKit knowledge is namespaced under agentic-dataset-factory, not polluting the global namespace — but it's still wrong content.

## Review Scope

1. **Duplicate knowledge assessment**: Determine if the two seed runs (2026-03-16 langchain-deepagents + 2026-03-17 default) created duplicate nodes/edges in FalkorDB, or if Graphiti's deduplication handled it
2. **Wrong content assessment**: Confirm that `guardkit graphiti seed` always seeds GuardKit system context (by design), and identify the correct command/approach for seeding project-specific architecture artefacts
3. **Missing architecture artefacts**: Confirm that the 9 ADRs, ARCHITECTURE.md, assumptions, domain-model, container, and system-context from `agentic-dataset-factory/docs/architecture/` were NOT seeded
4. **Remediation plan**: Determine whether to:
   - Clear the incorrectly seeded data and re-seed correctly
   - Leave the GuardKit system context (it may be useful) and additionally seed the project artefacts
   - Assess whether a `guardkit graphiti clear` or selective cleanup is needed
5. **Seeding command gap**: Identify whether a command exists (or needs to exist) for seeding project-specific architecture artefacts (ADRs, ARCHITECTURE.md, etc.) as opposed to GuardKit system context
